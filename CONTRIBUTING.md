# Contributing to Particule

Thank you for your interest in contributing to **Particule**. Contributions of all kinds are welcome, including bug reports, feature requests, documentation improvements, and pull requests. This guide explains how to contribute effectively.

---

## Code of Conduct
All contributors are expected to follow the project’s Code of Conduct. If the repository does not yet include one, please propose adding it through an issue or pull request.

---

## Before You Start

### Check Existing Issues
Before opening a new issue, review the existing list to ensure your report or request has not already been filed.

### Reporting Bugs
When reporting a bug, include:
- A clear and descriptive title  
- Steps to reproduce  
- Expected vs actual behavior  
- Environment information (OS, language/runtime version, dependencies)  
- Logs, screenshots, or minimal code samples if relevant  

### Requesting Features
When proposing a new feature:
- Explain the problem the feature solves  
- Describe how it should work  
- Note any alternatives you've considered  

---

## Contributing Code

### 1. Fork the Repository
Create your own fork, then create a new branch for your work. Use descriptive names such as:
- `feature/add-xyz`
- `bugfix/fix-abc`
- `docs/improve-readme`

### 2. Coding Standards
Follow the project’s existing style and structure. Apply formatters or linters relevant to the languages used in the project. Keep commits focused and readable.

### 3. Testing
If the project includes tests:
- Add tests for new functionality  
- Update tests when modifying behavior  
- Ensure all tests pass before submitting a PR  

If the project does not yet include tests, you may propose adding them.

### 4. Documentation
Update documentation where necessary, including:
- README sections  
- Code comments  
- API documentation  
- Examples or sample configs  

---

## Submitting Your Pull Request

When opening a pull request:
- Describe the purpose of your changes  
- Link related issues using keywords like `Fixes #12`  
- Provide details about testing the changes  
- Keep PRs focused on a single feature or issue when possible  

Maintainers may request modifications before merging.
